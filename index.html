Design System · Version 1.0.0

CME Bank Design System

A shared language for building CME Bank products — the tokens, components, and rules that keep onboarding, banking, and back-office surfaces feeling like one product. Everything here is accessible by default and themeable without duplication.

Overview

This system is organised in three layers. Work at the highest layer that solves your problem and you will rarely need to look below it.

LayerWhat it isWhen you touch it
Components Ready-made UI — buttons, fields, tables. Accessible, themed, state-complete. Every day. This is where product work happens.
Semantic tokens Named by intent — --text-primary, --action-danger-bg. When composing layouts or building something the library doesn't cover yet.
Primitives The raw palette and scales — --blue-400, --space-4. Almost never. Only when defining a new semantic token.
The one rule that keeps this system healthy

Components reference semantic tokens. Semantic tokens reference primitives. Never skip a layer — a component that reaches straight for --blue-400 cannot be themed, and will silently break the day the brand changes.

Getting started

Two stylesheets, in this order. tokens.css must come first — everything else resolves against it.

<link rel="stylesheet" href="tokens.css"> <link rel="stylesheet" href="components.css"> <button class="cme-btn cme-btn--primary">Open account</button>

Theming

Dark mode follows the operating system automatically. To override it, set data-theme on the root element — an explicit choice always wins over the OS preference.

<html data-theme="dark"> <!-- force dark --> <html data-theme="light"> <!-- force light --> <html> <!-- follow the OS -->
Fonts

The system uses Open Sans for customer-facing products and Roboto for back-office tooling. Load them in your page head; the token file falls back to the system UI font if they are unavailable, so nothing breaks — but the specimen below will not be accurate until they are loaded.

Token architecture

Every visual value in the system resolves through this chain. The example follows a single value from the raw palette to the pixel on screen.

TierTokenValueRule
1 · Primitive --green-500 #9CC53E Named for what it is. Never referenced by a component.
2 · Semantic --action-primary-bg → var(--green-500) Named for intent. Carries theming — this is the layer that changes per mode.
3 · Component .cme-btn--primary background: var(--action-primary-bg) Consumes semantics only. No hex, no primitives, no magic numbers.

Do

  • Use a component if one exists. Compose two before building a third.
  • Reach for semantic tokens when composing layouts.
  • Add a semantic token when the intent you need has no name yet.
  • Keep new tokens in tokens.css so they theme automatically.

Don't

  • Don't reference a primitive from a component or a product screen.
  • Don't write a hex value in product code — if it isn't a token, it isn't approved.
  • Don't create a component token that merely aliases a semantic one.
  • Don't encode a location in a name — --header-button-bg won't survive a redesign.

Why the label colour flips between them

The customer green is a light hue, so its label must be dark — white on it measures 2.56:1 and fails. The back-office blue is dark enough to carry white at 4.56:1. --action-primary-fg absorbs that difference, so the button component itself never knows which portal it is in.

Do

  • Set data-portal once, on <body>.
  • Use the same component classes in both portals.
  • Add new brand differences to the portal block in tokens.css.
  • Test every new component in both portals before shipping.

Don't

  • Don't fork a component to theme it — that is how one button became five.
  • Don't create .cme-btn--green or --blue. Brand is the portal's job.
  • Don't hardcode a typeface or radius in a component.
  • Don't nest one portal inside another except to demonstrate, as above.

Semantic tokens — use these, not the ramps

TokenPurposeContrast
--text-primaryBody copy, headings, table values8.1:1
--text-secondaryLabels, supporting copy, table headers5.2:1
--text-tertiaryHelper text and timestamps. Never for essential copy.4.3:1
--text-linkInline links. Must also carry an underline.6.7:1
--action-primary-bgThe one main action on a screen4.9:1
--action-danger-bgDestructive, irreversible actions only7.6:1
--action-accent-bgBack-office primary, selection, focus4.6:1

--border-subtleDecorative dividers only — carries no meaning—
--feedback-*-bg / -fgSuccess, warning, danger, info surfaces6.9:1+

Do

  • Pair a -bg token with its matching -fg token — those pairs are verified.
  • Reserve red, amber and teal for meaning. A teal button is not a "nice green".
  • Pair every status color with text or an icon.
  • Use --border-interactive for anything a user can act on.

Don't

  • Don't invent pairings across ramps — mixing an unverified fg/bg can fail contrast.
  • Don't use steps 400–600 of amber or teal as text colors; they are fills.
  • Don't rely on color alone to communicate state.
  • Don't use --border-subtle for input borders — it is below the 3:1 minimum by design.

Do

  • Use body-md as the default and step away only with reason.
  • One display-lg per view — it marks the page, not a section.
  • Keep running text near 65–75 characters per line.
  • Use overline for table headers and eyebrows.

Don't

  • Don't introduce sizes outside the scale — 13px and 18px are deliberately absent.
  • Don't go below 11px. It fails legibility at any weight.
  • Don't mix Open Sans and Roboto in one product surface.
  • Don't use uppercase for anything longer than a few words.

Control heights

Buttons, inputs and selects share one height scale so they line up on a row. Pick one size per row and stay with it.

Touch targets

Use --control-lg (44px) for any control that ships on mobile. WCAG 2.5.5 asks for 44×44px, and the smaller sizes do not meet it — they are for dense desktop tables and inline filters only.

Iconography

Icons are single-stroke, drawn on a 20×20 grid with a 1.6–1.8px stroke, and always inherit currentColor so they theme automatically.

check alert info close search download plus arrow lock user document delete
<svg width="16" height="16" aria-hidden="true"><use href="#i-check"/></svg>
SizeValueUse
--icon-sm16pxInside buttons, inputs, chips
--icon-md20pxStandalone icon buttons, nav
--icon-lg24pxFeature and status icons

Do

  • Add aria-hidden="true" when the icon sits beside a text label.
  • Give icon-only buttons an aria-label naming the action.
  • Let icons inherit currentColor so they theme with their parent.
  • Keep one icon per meaning across all products.

Don't

  • Don't use an icon alone to convey status — pair it with text.
  • Don't scale an icon to a size outside the three above.
  • Don't recolor an icon to a hue that carries a different meaning.
  • Don't rely on a tooltip as the only label for an icon button.

Why the primary button lightens on hover

The brand green is a light hue, so its label is dark. Darkening the background on hover would push the label below 4.5:1 — so the interaction ramp moves the other way. Rest, hover and pressed measure 4.9:1, 6.4:1 and 6.9:1 respectively: contrast improves as the user engages.

With icons

<button class="cme-btn cme-btn--tertiary" aria-label="Delete transaction"> <svg class="cme-btn__icon" aria-hidden="true"><use href="#i-trash"/></svg> </button>

Do

  • Lead the label with a verb — Save changes, Verify identity.
  • Keep one primary button per view. If two compete, one isn't primary.
  • Put a leading icon before the label; reserve trailing icons for direction.
  • Explain nearby what would enable a disabled button.

Don't

  • Don't write Submit, OK or Yes. Name the outcome.
  • Don't create a new variant for a one-off color.
  • Don't use --danger for anything reversible.
  • Don't put a spinner in place of the label — keep the label, swap the icon.

Field — floating label

The customer portal's field. The label starts as the placeholder and rises when the field is focused or filled, so the field never loses its label. Measurements come from the yup-comp: textinput component: 14px resting label at 40% opacity, 10px SemiBold at 80% once floated, 14px value, 44px tall.

<label class="cme-float"> <input class="cme-float__input" type="text" placeholder=" "> <span class="cme-float__label">Phone Number</span> </label>
placeholder=" " is required, not optional

The label floats using :placeholder-shown, which is how CSS knows the field is empty. The input needs placeholder=" " — a single space. Omit it and the label never rises. No JavaScript is involved.

One deliberate deviation from the Figma design

The design is fully borderless. A control with no visible boundary fails WCAG 1.4.11, which requires 3:1 contrast on the edge of an input — a low-vision user cannot find the field. A 1px underline at --field-underline (4.30:1) is the smallest change that satisfies it while keeping the open, boxless look. Set that token to transparent to restore the pure design; the component still works, but no longer meets 1.4.11.

Composed — country code and number

Do

  • Always pass placeholder=" " so the label can float.
  • Use this field throughout the customer portal.
  • Keep labels short — they must fit at 14px in the resting position.
  • Put the error message below the field, never inside it.

Don't

  • Don't use a real placeholder as a hint — that slot belongs to the label.
  • Don't use this field in the back-office; it uses the bordered field.
  • Don't remove the underline without accepting the 1.4.11 failure.
  • Don't slow the float past the built-in 140ms — it reads as lag.

Field — bordered

The back-office field, from the bo_input_field component. Denser than the customer field and boxed, because back-office screens pack many inputs into a grid where an explicit boundary reads faster than a floating label.

Disabled is not read-only

Disabled means "you can't edit this yet" — something must happen first. Read-only means "this value is informational and never editable". They look different on purpose: a read-only value drops its box entirely so it doesn't invite a click that will never work.

Select

Chooses one option from a known list. Shares the field anatomy with the text input — same height, padding, border and states.

<select class="cme-select" id="country"> <option>United Kingdom</option> </select>

Do

  • Use the native <select> wherever possible — it is accessible for free.
  • Add search once the list passes roughly ten options.
  • Order options meaningfully — frequency or alphabetically, not arbitrarily.
  • Match the menu width to the trigger.

Don't

  • Don't use a select for fewer than four options — radios show them all at once.
  • Don't nest submenus. Use a two-panel picker instead.
  • Don't rebuild a custom dropdown without the full ARIA combobox pattern.
  • Don't let hover move the keyboard focus position.
If you must build a custom select

You take on the whole ARIA combobox contract: role="combobox" with aria-expanded on the trigger, role="listbox" and role="option" in the list, arrow-key navigation, Enter to select, Escape to close and restore focus, and aria-activedescendant so DOM focus stays on the trigger. The native element gives you all of that already.

Checkbox & radio

Checkboxes select any number of independent options. Radios select exactly one from a set — and a set always has a default. Both use --selection-bg, bound to the product green rather than the back-office blue.

Why the tick is dark, not white

A white mark on the brand green measures 2.56:1 and fails. The selected state uses a green-900 mark on green-500 at 4.91:1 — the same dark-on-light rule the primary button follows, for the same reason.

<label class="cme-checkbox"> <input type="checkbox"><span class="cme-checkbox__box"></span> Paperless statements </label>

Do

  • Write labels as positive statements — Paperless statements, not No paper.
  • Make the whole label clickable by wrapping the input in it.
  • Always preselect a radio option.
  • Group related choices in a <fieldset> with a <legend>.

Don't

  • Don't use a checkbox where only one option can apply — that's a radio.
  • Don't use radios for on/off — that's a switch.
  • Don't leave a radio group with nothing selected.
  • Don't apply a change immediately from a checkbox inside a form with a save button.

Switch

Turns a setting on or off, applying immediately. If the change needs a save button, use a checkbox instead.

<label class="cme-switch"> <input type="checkbox" role="switch" checked><span class="cme-switch__track"></span> Two-factor authentication </label>
Switch or checkbox?

A switch takes effect the moment it is flipped. A checkbox states an intention that is committed later, when the form is submitted. Choosing the wrong one is the most common source of "did that save?" confusion.

Chip & badge

A compact, read-only marker for status or category. Every chip pairs a color with a word — the color reinforces the label, it never replaces it.

Draft Verified Pending review Rejected In progress
<span class="cme-chip cme-chip--success"><span class="cme-chip__dot"></span> Verified</span>

Do

  • Keep labels to one or two words.
  • Use one consistent chip colour per status across every product.
  • Include the dot — it helps at a glance in dense tables.
  • Use sentence case.

Don't

  • Don't make chips clickable. If it acts, it's a button.
  • Don't use a bare colored dot with no text.
  • Don't invent new chip colours for new statuses — map to the four meanings.
  • Don't stack more than three chips in one table cell.

  • State what happened, then what to do about it.
  • Place the alert next to what it refers to.

  • Don't apologise. Explain the problem and the fix.
  • Don't stack more than one alert per region.
  • Don't use danger styling for a warning — it dilutes real errors.
  • Don't auto-dismiss a message the user must act on.

  • Don't combine zebra striping with row selection.
  • Don't right-align text or centre anything but icons.
  • Don't paginate below about 50 rows — scrolling is cheaper for the reader.

Accessibility

These are commitments, not aspirations. Each is testable, so compliance is something you can demonstrate rather than claim. The components on this page already meet them.

On disabled controls

Disabled text sits at 3.5:1, below the 4.5:1 body minimum. This is deliberate and permitted — WCAG 1.4.3 exempts inactive controls, and dimming is how "unavailable" reads. What the exemption does not cover is using the disabled colour for ordinary secondary text: use --text-secondary or --text-tertiary for that.

Do

  • Test with the keyboard alone before you ship.
  • Give every form control a programmatically associated label.
  • Announce dynamic changes with a live region.
  • Respect the user's motion and contrast preferences.

Don't

  • Don't use placeholder as a label.
  • Don't trap keyboard focus inside a component.
  • Don't convey state with colour alone.
  • Don't set a positive tabindex — it breaks the natural order.

Contributing

How something new gets into the system, and how it changes once it's here.

Adding a component

StepWhat's required
1 · SearchShow which existing components you reviewed and why each fell short.
2 · ProposeThe use case, the variants needed, and which products will consume it.
3 · ProveTwo independent teams need it. Until then it lives locally in your product.
4 · BuildAll states covered, semantic tokens only — no hex values, no primitives.
5 · DocumentAnatomy, do/don't, and the accessibility contract, before it ships.
6 · ReviewThe system owner signs off on naming and token usage.
The rule of two

Step 3 is the one that keeps this library from sprawling. A component earns a place in the shared system when a second team needs it. Before that, it is product-local work — and most one-off components never find a second consumer.

Naming

Deprecation

Nothing is deleted without notice. Every retirement follows the same four steps: mark the token or component as deprecated with a pointer to its replacement, announce it in the changelog, keep both working for one minor version, then remove it at the next major with a migration note listing every mapping.
